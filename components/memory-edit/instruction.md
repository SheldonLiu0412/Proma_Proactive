## 引导用户规范操作需求

当用户表达修改意图时，帮助用户转化为好的规范操作需求；**对于用户模糊、信息不足的需求，你必须主动追问、引导用户补齐必填字段后再执行**。

好的操作需求应包含以下维度：

| 维度 | 说明 | 示例 |
|------|------|------|
| **操作类型** | 增/删/改 | "添加"、"更新"、"删除" |
| **目标** | 修改哪里 | "画像里的技术偏好"、"corr_002"、"某个 SOP" |
| **内容** | 改成什么 | 具体的新内容描述 |

### 各操作类型的必填字段

不同类型操作的必填字段不同。如果用户需求缺少以下关键信息，**不要直接执行**，先追问补齐：

| 操作类型 | 必填字段 | 追问示例 |
|----------|----------|----------|
| **画像更新** | 段落位置 + 内容 | "你想加到画像的哪个部分？具体写什么？" |
| **纠偏记录 (add)** | type + summary + detail | "这条纠偏属于哪类？（agent-behavior / skill-update / user-preference）简短摘要是什么？详细说明是什么？" |
| **纠偏记录 (edit)** | 目标 ID + 修改内容 | "你想修改哪条 correction？改 summary 还是 detail？改成什么？" |
| **纠偏记录 (delete)** | 目标 ID | "你想删除哪条 correction？可以用内容描述，我帮你匹配 ID。" |
| **SOP 删除** | 目标 ID | "你想删除哪个 SOP？" |

**纠偏记录字段说明**：
- `type`：分类 —— agent-behavior（Agent 行为纠正）、skill-update（Skill 内容纠正）、user-preference（用户个人偏好）
- `summary`：一句话摘要，用于快速识别这条纠偏的主题
- `detail`：详细说明，提供具体场景、示例或背景

### 列出记忆目录结构（仅供读取）

在操作前，你可以先查看当前记忆的完整目录结构，以便用户需要读取记忆原文的时候使用：

```bash
cd {{PROJECT_ROOT}}
npx tsx src/scripts/memory-edit.ts --list
```

该命令会输出 `.memory/` 目录的完整树形结构，包含所有文件和大小。

### 操作执行流程

在执行修改前，**应先向用户复述具体最终修改需求并确认**，用户确认后，执行操作脚本：

```bash
cd {{PROJECT_ROOT}}
npx tsx src/scripts/memory-edit.ts --instruction "用户的具体操作需求"
```

脚本内部会：分析操作需求 → 输出结构化操作计划 → 映射到具体操作 → 返回操作结果

**把脚本的输出返回给用户**。

### Memory 系统支持修改的记忆类型

| 类型 | 说明 | 执行方式 | 好的示例需求 |
|------|------|----------|--------------|
| **画像更新** | 修改 profile.md | 脚本返回具体指导，**你使用 Edit 工具手动编辑** | "在画像的技术偏好部分加上：我主力用 TypeScript，习惯英文注释" |
| **纠偏记录 (add)** | 添加 correction | 脚本直接修改 | "添加一条 user-preference 纠偏：summary 是"返回代码时不解释每行"，detail 是"用户习惯直接看代码，不需要逐行注释说明"" |
| **纠偏记录 (edit)** | 修改 correction | 脚本直接修改 | "把 corr_002 的 summary 改成"简短词直接继续"，detail 不变" |
| **纠偏记录 (delete)** | 删除 correction | 脚本直接修改 | "删除那条讲不要催复的 correction" |
| **SOP 删除** | 删除 SOP 候选 | 脚本直接删除 | "删除 sop_001" |

### Profile 编辑指导

当脚本返回 profile 编辑指导时，你会收到：
- profile.md 的完整文件路径
- 用户画像写作规范（视角、结构、风格要求）
- 建议的目标段落和内容

请使用 Edit 工具直接编辑该文件，**不允许重新创建或删除 profile.md**。
